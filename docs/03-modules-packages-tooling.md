## Getting Started

### Install

```bash
# From source (development)
git clone https://github.com/IT-KYNERA/KYLE
cd kl
cargo build --workspace
# Binary: target/debug/kl
```

### Create a Project

```bash
kl new myapp
cd myapp
kl run
```

`kl new` creates:

```
myapp/
├── src/main.kl           — entry point with args:[str]
├── tests/test_main.kl     — test stub
├── kl.toml                — project manifest
└── .gitignore             — target/, *.klc-build/, kl.lock
```

### CLI Reference

| Command | Description |
|---------|-------------|
| `kl new <project>` | Create new project (src/, tests/, kl.toml, .gitignore) |
| `kl build [<file.kl>]` | Compile to native binary (project mode if no file) |
| `kl run [<file.kl>]` | Compile and execute |
| `kl check <file.kl>` | Type-check without codegen |
| `kl parse <file.kl>` | Parse and dump AST |
| `kl mir <file.kl>` | Parse and dump MIR |
| `kl fmt <file.kl>` | Format source code (in-place) |
| `kl test` | Run project tests (type-check tests/ directory) |
| `kl add <dep@version>` | Add dependency to kl.toml |
| `kl remove <dep>` | Remove dependency from kl.toml |
| `kl info` | Show project info |
| `kl lsp` | Start language server (stdio, for editor integration) |
| `kl help` | Show usage |
| `kl --version` | Show version |

**Flags:**
- `--release` — build/run with release output (`target/release/`)
- `--check` — with `fmt`: check if formatting differs without writing

**Binary names:** `kl` (primary) and `klc` (legacy alias) — both compiled from
the same source, auto-detect the name from `argv[0]`.

## Module System

### Imports

```kl
import io                          # module import
import math from "std/math"        # explicit path
from io import println             # specific import
from io import println as pln      # aliased import
```

### Visibility

Kyle uses naming conventions instead of keywords:

| Convention | Meaning |
|-----------|---------|
| `Public` | Public (default) |
| `_protected` | Protected (subclasses + same module) |
| `__private` | Private (same module only) |
| `UPPERCASE` | Constant (immutable, compile-time) |

No `export`, `pub`, `private` keywords — visibility is by naming.

### Module Resolution (current)

The module resolver (`klc_driver/src/pipeline.rs`) searches:
1. Relative to the source file's directory
2. The `std/` directory in the project root

**Limitations (planned for Phase 9+):**
- No nested module paths (`from .user import User` — relative imports)
- No package-qualified imports (`from mypackage.utils import helper`)
- Module path resolution is flat (no `std/net/tcp.kl` hierarchy)

## Package Manager

### Manifest (kl.toml)

```toml
name = "myapp"
version = "0.1.0"
edition = "1"
authors = []
license = "MIT"
description = "A Kyle project"

[compiler]
optimization = "O2"
target = "native"

[dependencies]
# dependency_name = "version"
```

### Lock File (kl.lock)

Auto-generated by `kl build`. Records exact versions of dependencies.

### Commands

| Command | Status | Description |
|---------|--------|-------------|
| `kl new` | ✅ | Create project structure |
| `kl add dep@version` | ✅ | Add dependency to manifest |
| `kl remove dep` | ✅ | Remove dependency |
| `kl info` | ✅ | Show project metadata |
| `kl build` | ✅ | Compile project (reads src/main.kl) |

## Build System

### Project Mode

When `kl build` or `kl run` is run without a file argument, the CLI:
1. Searches upward for `kl.toml` (project root)
2. Reads `src/main.kl` as the entry point
3. Compiles to `target/debug/main` (or `target/release/main`)
4. For `kl run`: executes the binary with forwarded args

### Single-File Mode

When `kl build file.kl` is given a file path:
1. Compiles that file directly
2. Output goes to `target/debug/<filename_stem>`
3. No manifest required

### Build Artifacts

```
target/
├── debug/
│   ├── main          — native binary
│   ├── main.ll       — LLVM IR (debug)
│   └── main.o        — object file
└── release/
    └── main          — native binary (optimized)
```

Build artifacts are in `target/`, never in the source directory.

## VS Code Extension

Install from `.vsix`:

```bash
code --install-extension vscode-kl/kl-0.2.0.vsix
```

Features:
- Syntax highlighting (TextMate grammar)
- LSP client (launches `kl lsp`)
- Commands: `KL: Run`, `KL: Build`, `KL: Check`
- Snippets (20 common patterns)
- Dot-triggered autocompletion (struct/class/enum/str/list/dict fields + methods)
- Scope-aware completion (params, auto-declared vars, block-local vars)

Configuration:
- `kl.klcPath` — path to the `kl` binary (default: `"kl"`)
